# ZC300 系列运动控制器上位机软件开发技术报告

## 0. 文档概述
本报告旨在为 ZC300 系列运动控制器的上位机软件开发提供完整的技术指导。文档基于《ZC300-ModbusRTU通信协议-用户手册》、《ZC300控制器软件使用说明书》及《ZC300系列运动控制器使用说明书》汇编而成，主要涵盖通信物理层、Modbus RTU 协议实现、核心操作码指令、物理单位换算逻辑、运动控制流程、异常错误处理及上位机软件架构建议。

> **注意**：本报告**不含 Modbus 寄存器表**（寄存器映射表已单独提供），重点阐述功能实现逻辑与指令通信细节。

---

## 1. 系统通信架构与物理层

### 1.1 通信方式
ZC300 系列控制器支持 **USB（虚拟串口）**、**RS-232** 和 **RS-485** 三种通信物理层，底层均采用 **Modbus RTU** 协议。

*   **USB 通信**：通过 FTDI 芯片将 USB 转换为串口，上位机需通过 Windows 设备管理器获取对应的 COM 端口号，并实现端口扫描机制以自动发现控制器。
*   **RS-485 通信**：采用**半双工**接线，支持多从站组网（最多 255 个节点）。构建 RS-485 网络时，*建议在总线最后一个从站设备的 A、B 线之间并联一个 120 欧姆的终端电阻*，以保证通信质量。

### 1.2 串口通信参数
Modbus RTU 采用主从问答机制，上位机作为**主站**，控制器作为**从站**。通信参数的配置具有严格要求，所有网络设备必须一致：

| 参数项 | 设定值 | 备注 |
| :--- | :--- | :--- |
| **波特率** | `115200` | 出厂固定，不可更改 |
| **数据位** | `8` | |
| **停止位** | `1` | |
| **校验位** | `无 (None)` | |
| **协议模式** | `RTU` | 二进制数据，CRC 校验 |

### 1.3 主从通信机制
*   所有通信均由**上位机（主站）发起查询**。
*   控制器（从站）响应时，会根据指令执行动作并返回正常的响应报文，或返回异常响应报文。
*   在 RS-485 组网中，每个从站必须具备唯一的 `从站地址`（默认地址为 `1`，可通过面板或协议配置，范围 1~255）。

## 2. 通信协议与操作码

### 2.1 支持的 Modbus 功能码
上位机软件需实现以下四个 Modbus 功能码，用于寄存器的读写操作：

| 功能码 (Hex) | 名称 | 用途 |
| :--- | :--- | :--- |
| `0x03` | **读取保持寄存器** | 读取各类设置参数、输出状态、设备信息 |
| `0x04` | **读取输入寄存器** | 读取 IO 输入状态、软件版本、只读状态位 |
| `0x06` | **写单个寄存器** | 修改单个寄存器的数值（如单位、地址、蜂鸣器） |
| `0x10` | **写多个寄存器** | 写入连续参数，**也是执行操作码指令的唯一方式** |

### 2.2 核心操作码指令（运动指令）
**操作码原理**：核心的控制指令并不写入特定的参数寄存器，而是通过向命令寄存器（即 `30050`）写入特定的操作码，控制器识别后立即执行对应的动作。

*   **规定**：操作码指令**仅支持** Modbus 功能码 `0x10`（写多个寄存器），需要严格按照 `操作码(30050)` + `参数1(30051)` + `参数2(30052)` + `参数3(30053)` 的顺序连续写入。
*   **地址换算注意**：Modbus 协议报文中的起始寄存器地址 = 手册十进制地址 - 1。
    *   例：寄存器 `30050`，报文中的实际地址为 `30050 - 1 = 30049 = 0x7561`。

**具体的操作码指令表如下**（所有数值均为十六进制）：

| 指令类型 | 操作码 (30050) | 参数1 (30051) | 参数2 (30052) | 参数3 (30053) | 功能说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **绝对位置运动** | `0x0064` | `轴号` | `方向` | `×` | 运动到设定好的绝对目标坐标 |
| **定长运动** | `0x0065` | `轴号` | `方向` | `×` | 正向/反向运动设定的定长距离 |
| **连续运动** | `0x0066` | `轴号` | `方向` | `×` | 持续运动，直到收到停止指令或碰触限位 |
| **减速停止** | `0x0067` | `轴号` | `×` | `×` | 按照设定的减速度（加速度值）减速停机 |
| **立即停止** | `0x0068` | `轴号` | `×` | `×` | 紧急制动，立即停机 |
| **归零命令** | `0x0069` | `轴号` | `×` | `×` | 触发执行归零流程 |
| **恢复出厂** | `0x006C` | `×` | `×` | `×` | 恢复全部参数至出厂默认值 |
| **保存参数** | `0x006D` | `×` | `×` | `×` | **将修改的参数写入 Flash，掉电保存** |

> **参数取值说明**：
> *   **轴号**：`0x30`（XYZ 三轴联动）、`0x31`（X 轴）、`0x32`（Y 轴）、`0x33`（Z 轴）
> *   **方向**：`0x50`（P，表示正向/前进）、`0x4E`（N，表示负向/后退）
> *   **表格中“×”符号**：表示该位置无需写入有效数据，但在功能码 `0x10` 报文中仍需占位，通常补 `0x00`。

## 3. 数据类型与单位换算

上位机在读写数据时，需要根据寄存器的定义使用正确的数据类型。控制器中的数据地址基本遵循“高位在前（大端模式）”的规则。

### 3.1 数据类型规范
| 数据类型 | 占用寄存器数 | 说明 |
| :--- | :--- | :--- |
| **SHORT (16位整数)** | 1 | 状态位、使能状态、IO状态、简单枚举 |
| **LONG (32位整数)** | 2 | 每转脉冲数、设备编号 |
| **FLOAT (32位浮点数)** | 2 | 位置坐标、速度（初/常/加）、丝杠导程、传动比 |

**浮点数示例**：
浮点数 `15.0` 的 IEEE 754 十六进制大端表示为 `0x41700000`。
当使用功能码 `0x10` 写入常速度寄存器 `30129`（高字节）和 `30130`（低字节）时，报文中数据顺序应为：`41 70 00 00`。

### 3.2 物理单位与脉冲转换逻辑
上位机在输入、输出和显示坐标/速度时，需依据用户设定的单位（`PP`、`mm` 或 `deg`）进行实时换算。当单位切换为 `mm` 或 `deg` 时，上位机必须以当前的机械参数为基准进行计算。

**1. 脉冲当量（控制精度）计算**
*   **平移台（丝杠传动）**：
    脉冲当量 (mm/Pulse) = 丝杠导程 (mm/r) / 每转脉冲数 (Pulse/r)
*   **旋转台（传动传动）**：
    脉冲当量 (deg/Pulse) = 360° / (每转脉冲数 (Pulse/r) × 传动比)

**2. 物理量与脉冲数之间的实际换算公式**
*   **物理位置转换**：
    实际位移量 (mm 或 deg) = 目标脉冲数量 × 脉冲当量
    目标脉冲数量 = 实际位移量 (mm 或 deg) / 脉冲当量
*   **物理速度转换**：
    实际速度 (mm/s 或 deg/s) = 设定速度 (PP/s) × 脉冲当量
    设定速度 (PP/s) = 实际速度 (mm/s 或 deg/s) / 脉冲当量

**开发建议**：上位机软件在初始化连接时，建议读取并缓存**丝杠导程**、**传动比**、**每转脉冲数**三个关键参数，以便在显示和操作时，CPU 能够精确地进行毫米/度与脉冲数量之间的内部数学转换，减少因轮询造成的通信冗余。

## 4. 运动控制核心逻辑

### 4.1 运动控制执行流程
上位机在发送任何运动指令前，应遵循以下基本流程，以避免触发控制器的异常报警：
1.  **检查使能状态**：读取目标轴的使能状态寄存器（如 X 轴的 `30066`）。若返回 `0x00`（未使能），则弹出提示并拒绝发送运动指令。
2.  **检查限位与急停**：读取状态寄存器 `30015`，解析位映射，检查是否触发了正/负限位，或急停按钮是否被按下（物理急停按下时，上位机发送运动指令将返回异常码 `0x08`）。
3.  **写入运行参数（可选）**：如果需要更改速度或距离，先通过 `0x10` 向对应的寄存器写入目标值。
4.  **发送操作码**：利用功能码 `0x10` 向寄存器 `30050` 发送运动操作码指令。

### 4.2 状态监控与位置回读
上位机需要使用**定时轮询机制**（建议周期 50ms ~ 100ms），实时更新软件界面的数据和状态指示灯。需监控的项目包括：
*   **轴物理位置（只读）**：通过功能码 `0x04` 读取输入寄存器 `30016`(X轴)、`30018`(Y轴)、`30020`(Z轴)。返回值为浮点数，需直接转换为物理单位显示在面板上。
*   **运动状态（只读）**：通过 `0x04` 读取 `30012`(X轴)、`30013`(Y轴)、`30014`(Z轴)。`0x01` 表示正在运动中，`0x00` 表示停止。
*   **限位与报警状态（只读）**：通过 `0x04` 读取 `30015` 寄存器。上位机需按位解析：
    *   Bit 0~2：X轴正/负限位、原点
    *   Bit 3~5：Y轴正/负限位、原点
    *   Bit 6~8：Z轴正/负限位、原点
    *   Bit 9：急停状态
    *   Bit 10~12：X/Y/Z 轴驱动器报警状态

### 4.3 参数保存机制（重要）
参数保存是上位机开发中的关键易错点。
*   **如何保存**：上位机设置完丝杠导程、速度、归零模式等参数后，参数仅暂存在控制器的 RAM 中。若要断电重启后生效，必须向 `30050` 发送操作码 `0x006D`。
*   **保存失败的情况**：当上位机发送 `0x006D` 保存指令时，如果三轴中**任意一根轴正在运动**，控制器将**拒绝执行**，并返回异常码 `0x04`（存储失败）。
*   **开发建议**：在 UI 界面上提供“保存参数”按钮，点击该按钮的软件触发逻辑应为：先调用 `0x03` 读取三根轴的运动状态寄存器，若全部确认处于停止状态，再发送保存指令。

## 5. 异常响应与容错机制

上位机软件必须具备完善的异常响应处理能力。当控制器无法执行指令时，会返回一个**异常响应**。上位机需解析该异常并给予用户明确的提示。

### 5.1 异常报文体
*   异常功能码 = 请求功能码 + `0x80`（例如：发送 `0x04` 读指令，收到 `0x84` 表示异常）。
*   异常数据域包含 **1 个字节的异常码**。

### 5.2 错误代码表
上位机开发者需在软件中预置错误代码映射表，用以弹窗提示用户：

| 异常码 (Hex) | 错误名称 | 含义说明与应对策略 |
| :--- | :--- | :--- |
| `0x01` | 功能码错误 | 控制器不支持该功能码（通常是协议实现的 Bug） |
| `0x02` | 寄存器地址错误 | 尝试读取或写入不存在的地址，或写入了只读寄存器 |
| `0x03` | 数据长度错误 | 发送的 Modbus 报文长度计算错误 |
| **`0x04`** | **存储失败** | 发送保存参数指令时，某轴正在运动导致数据无法写入 Flash |
| `0x05` | 数据错误 | 写入的数据值超出了该参数的物理允许范围 |
| **`0x06`** | **指令报警** | 发送运动指令时，目标轴**正在运动**，需先发送停止指令 |
| **`0x07`** | **指令报警** | 发送运动指令时，当前轴已触发**限位**，只能向反方向移动 |
| **`0x08`** | **指令报警** | 控制器面板的红色急停按钮**被按下**，需物理复位急停 |
| **`0x09`** | **指令报警** | 发送运动指令时，目标轴处于**未使能**（失电）状态 |
| `0x0A` | 指令报警 | 操作码错误，上位机发送了未知的 `30050` 操作码值 |

## 6. 上位机软件架构设计建议

基于官方说明书的软件逻辑，上位机软件推荐采用“驱动层-控制层-界面层”三层架构。

### 6.1 连接模块（驱动层）
*   **串口自动扫描**：遍历 COM1~COM255，尝试发送读取设备型号的指令（功能码 `0x04`，读取 `30001`，长度 `7`）。解析响应的字符串（如 `5A 43 33 30 30...` 对应 "ZC300"）以识别有效控制器。
*   **多设备管理**：根据扫描结果，在软件左侧或顶部提供控制器列表/标签页，支持 RS-485 网络下的多设备控制。

### 6.2 状态控制面板（控制层与界面层）
*   **位置反馈**：循环轮询浮点数位置并实时更新。用户需直接输入以 mm 或 deg 为单位的目标值。
*   **单位自动适配**：上位机需要读取 `30072` 等寄存器并在界面动态切换坐标系（PP / mm / deg）。
*   **轴使能开关**：UI 提供拨动开关，点击时发送 `0x06` 修改 `30066` 寄存器的值（使能/去使能）。
*   **急停功能**：界面提供醒目的红色“急停”按钮，点击后发送 `0x0068` 操作码对所有运动轴进行立即停机。

### 6.3 运行方案控制（自动化模块）
当需要执行复杂的自动化流程时，上位机需要实现一个**脚本解释器**，模拟官方软件中的“运行方案控制”：
*   **流程控制轴**：配置流程控制轴数组，解释器在向下执行下一条指令前，必须确认数组中的所有物理轴都处于“空闲（停止）”状态。
*   **循环与条件跳转**：实现堆栈机制，支持 `循环开始/结束` 指令（支持无限循环或指定次数循环），以及状态等待指令。
*   **指令队列**：将连续的运动指令（定长、绝对位置、连续）放入队列中，顺序下发。

## 7. 开发附录与注意事项

### 7.1 CRC 校验计算规则
*   **校验域**：从站地址位到数据位结束（不包含 CRC 本身）。
*   **算法**：标准的 Modbus CRC-16 算法。
*   **字节序**：发送时，**CRC 校验的低字节在前，高字节在后**。
    *   例：计算出来的 CRC 值为 `0xABCD`，报文尾部应为 `CD AB`。

### 7.2 浮点数与数据类型转换
由于 Modbus 协议通信中，32位浮点数（FLOAT）占用 2 个寄存器，且为**大端模式（高位在前）**传输，因此在 C# 或 Python 等常见上位机开发语言中需注意字节序的转换处理。

*   **Python 处理**：
    在 Python 中，可以使用 `struct` 模块的 `unpack` 方法，指定大端浮点数格式 `>f` 来解码两个寄存器传输回来的四个字节。例如：解码字节流 `0x41, 0x70, 0x00, 0x00`，得到的浮点数结果即为 `15.0`。

*   **C# 处理**：
    在 C# 中使用 `BitConverter.ToSingle` 时需注意平台字节序。由于 Modbus 传输为大端模式，而 Windows 系统通常为小端序，因此在调用转换函数前，需判断系统是否为小端序，若是则需将字节数组翻转后，再转换得到正确的浮点数。

### 7.3 寄存器地址使用提示
虽然您已获得寄存器表，但务必注意一个极易出错的地方：
在构造 `0x06` 或 `0x10` 的 Modbus RTU 报文时，**寄存器地址必须在手册给出的十进制地址基础上减 1**。
*   **错误做法**：报文起始地址直接写 `0x7562`（即 30050 的十六进制）。
*   **正确做法**：30050 对应的地址 = `30050 - 1 = 30049 = 0x7561`。即 Modbus 报文中地址应写 `0x75 0x61`。
